# 多模态 RAG 项目 — 后端技术分析文档

> 本文档仅分析后端技术架构，不涉及前端。基于当前代码库梳理，供架构评审与二次开发参考。

---

## 一、项目概述与定位

- **项目名称**：AI 多模态智能问答助手（RAG API）
- **定位**：企业级多模态 RAG（检索增强生成）系统，支持文本/图片/文件/视频附件、知识库检索、实时联网、MCP 工具、浏览器助手与电脑管家等扩展能力。
- **后端技术**：Python 3.x + FastAPI 异步 Web 框架，兼容 MySQL / PostgreSQL / SQLite；向量库支持 Zilliz Cloud（Milvus 兼容）与 Qdrant；对象存储为 MinIO（S3 兼容）；任务队列为 Celery + Redis。

---

## 二、技术栈总览

| 类别 | 技术选型 | 说明 |
|------|----------|------|

## 三、应用架构

### 3.1 入口与启动

- **主入口**：`backend/app/main.py`，通过 `lifespan` 管理启动/关闭。
- **启动脚本**：`backend/run.py`，负责切到 `backend` 目录、设置 Windows 下 `ProactorEventLoop`（兼容 Playwright 等子进程），再启动 Uvicorn。
- **启动时**：
  - 初始化日志（`setup_logging`）。
  - 设置 asyncio 异常处理器，减少 Windows 下 Playwright 子进程未捕获异常刷屏。
  - 创建数据库表（`Base.metadata.create_all`），并可选执行 `messages.attachments_meta` 列存在性及 LONGTEXT 修正（MySQL）。
- **关闭时**：释放数据库引擎（`engine.dispose()`）。

### 3.2 中间件与全局行为

- **CORS**：`CORSMiddleware`，允许源由 `settings.CORS_ORIGINS` 配置。
- **GZip**：`GZipMiddleware`，最小压缩大小 1000 字节。
- **请求 ID**：每个请求生成或透传 `X-Request-ID`，写入 `request.state.request_id`，并在响应头中回传，便于链路追踪与审计。
- **异常处理**：对 `StarletteHTTPException`、`RequestValidationError` 及未捕获 `Exception` 统一 JSON 格式，并处理 `detail` 可能为 bytes 的序列化问题；响应中可带 `request_id`。

### 3.3 路由挂载

- 所有 v1 API 挂载在 `/api/v1` 下，由 `app/api/v1/__init__.py` 中的 `api_router` 聚合子路由：
  - `/auth` — 认证（注册、登录、当前用户、改密等）
  - `/dashboard` — 仪表盘统计与用量快照
  - `/files` — 文件上传、列表、删除
  - `/knowledge-bases` — 知识库 CRUD、添加/移除文件、检索、异步任务触发
  - `/tasks` — 异步任务状态查询（轮询 task_id）
  - `/search` — 以文搜图、图搜图、多模态统一检索
  - `/chat` — 问答（同步/流式）、附件上传、会话与消息列表、删除会话
  - `/billing` — 计费相关（套餐、订阅、订单、发票等）
  - `/audit-logs` — 审计日志查询
  - `/mcp-servers` — MCP 服务配置 CRUD
  - `/steward` — 浏览器助手
  - `/computer-steward` — 电脑管家
  - `/bash` — Bash 命令审批（白名单、审批模式）

根路径 `/` 与 `/health` 在 `main.py` 中直接定义；`/health` 会检测数据库、Redis、向量库、MinIO 连通性。

## 四、核心配置（app/core/config.py）

配置通过 `pydantic_settings.BaseSettings` 从项目根目录 `.env` 读取，大小写不敏感，未声明字段忽略。

### 4.1 主要配置项摘要

- **API**：`API_V1_STR`、`PROJECT_NAME`、`CORS_ORIGINS`。
- **数据库**：`DATABASE_URL`（异步连接串）。
- **Redis**：`REDIS_URL`；缓存开关 `CACHE_ENABLED`、各场景 TTL（统计/列表/会话/详情）。
- **Celery**：`CELERY_BROKER_URL`、`CELERY_RESULT_BACKEND`，未填则与 `REDIS_URL` 一致。
- **向量库**：`VECTOR_DB_TYPE`（zilliz | qdrant）、Zilliz 的 URI/Token/集合名/维度、Qdrant URL/API Key。
- **MinIO**：Endpoint、Access/Secret、Bucket、Secure。
- **安全**：`SECRET_KEY`、`JWT_SECRET_KEY`、`JWT_ALGORITHM`、Token 过期时间。
- **AI 模型**：`OPENAI_API_KEY`、`OPENAI_BASE_URL`、`EMBEDDING_MODEL`、`LLM_MODEL`、`VISION_MODEL`、`RERANK_MODEL`、`OCR_MODEL`；阿里云百炼 `DASHSCOPE_API_KEY`、`DASHSCOPE_BASE_URL`、`USE_DASHSCOPE`；本地模型相关（可选）。
- **文件上传**：最大体积、允许类型、禁止扩展名、同 MD5 策略（use_existing/overwrite）、扫描版 PDF OCR 阈值与 DPI、文件名长度、病毒扫描（ClamAV）、敏感信息脱敏、审计开关。
- **智能问答附件**：单条消息附件数量/大小限制、图片与文件类型、视频扩展名、上传缓存 TTL。
- **对话历史**：最大会话数、每页条数、上下文消息条数、超出部分是否用 LLM 总结。
- **联网检索**：是否启用、最大结果数。
- **RAG**：置信度阈值、RRF K、是否 BM25、查询改写与子问题数、上下文窗口扩展、以文搜图时是否扩展同义词。
- **分块**：默认 chunk_size、chunk_overlap、最大扩展比例。
- **LangChain**：是否使用 LangChain 封装 LLM/RAG/Agent。
- **记忆**：本地记忆开关与 DB 路径（OpenClaw 风格）。
- **Bash**：是否允许、超时、输出截断、安全命令白名单、审批模式与过期时间、PTY。
- **限流**：每日上传/对话上限、检索 QPS、总开关。

## 五、数据层

### 5.1 关系型数据库（SQLAlchemy 2.x 异步）

- **引擎**：`app/core/database.py`，根据 URL 是否含 `sqlite` 选择 NullPool 或连接池；`pool_pre_ping=True`。
- **会话**：`AsyncSessionLocal`，`get_db()` 为 FastAPI 依赖，请求内 yield 会话并在 finally 中关闭。
- **Celery 内使用**：任务内通过 `create_async_engine_and_session_for_celery()` 在当前 event loop 下创建新的 engine 与 session 工厂，使用 NullPool，避免 “Future attached to a different loop”。

### 5.2 领域模型（app/models）

| 模型 | 表名 | 说明 |
|------|------|------|
| User | users | 用户：用户名、邮箱、密码哈希、角色、套餐、积分、最后登录等 |
| File | files | 文件：归属用户、文件名、类型、大小、存储路径、MD5、状态（uploading/processing/completed/failed）、分块数 |
| KnowledgeBase | knowledge_bases | 知识库：名称、描述、分块策略、embedding/llm 模型、temperature、rerank/混合检索开关 |
| KnowledgeBaseFile | knowledge_base_files | 知识库-文件关联 |
| Chunk | chunks | 文档块：内容、索引、起止字符、metadata、vector_id（与向量库对应） |
| Conversation | conversations | 对话：用户、可选知识库、标题、时间 |
| Message | messages | 消息：角色、内容、tokens、模型、RAG 置信度/检索上下文/来源、工具调用、联网内容、附件 meta（LONGTEXT） |
| UsageRecord | usage_records | 使用记录：类型、资源、数量、单位、成本 |
| Plan | plans | 套餐：名称、价格、月额度、特性 JSON |
| Subscription | subscriptions | 用户订阅 |
| Order | orders | 订单 |
| Invoice | invoices | 发票 |
| AuditLog | audit_logs | 审计：操作类型、资源类型/ID、详情、IP、request_id |
| McpServer | mcp_servers | MCP 服务：名称、传输类型（stdio/streamable_http/sse）、JSON 配置、启用状态 |

MySQL 下 `messages.attachments_meta` 使用 LONGTEXT，以容纳含 base64 图片的 JSON。

### 5.3 向量数据库

- **抽象**：`app/services/vector_store.py`，`get_vector_client()` 返回单例的 Zilliz 或 Qdrant 客户端。
- **Zilliz**：基于 pymilvus `MilvusClient`，集合名与维度来自配置，COSINE 度量；支持 `ensure_collection`、`insert`、`search`、按 chunk_id 列表删除。
- **Qdrant**：qdrant-client，等效能力。
- **chunk 与向量 ID**：`chunk_id_to_vector_id(chunk_id)` 为确定性映射，便于插入与删除时一致。

### 5.4 Redis

- **用途**：限流计数（按用户/按天/按秒）、缓存（会话列表/详情、知识库列表/详情、仪表盘统计、智能问答上传临时数据）、Celery Broker/Backend。
- **缓存 Key**：通过 `cache_service` 统一前缀与 TTL（见 config 中 CACHE_TTL_*）。

### 5.5 MinIO（S3 兼容）

- **用途**：用户上传文件与知识库文件的持久化存储；由 `FileService` 在 upload 时写入，路径存于 `files.storage_path`。
- **健康检查**：`/health` 中检查 bucket 是否存在。

### 5.6 Celery

- **应用**：`app/celery_app.py`，broker/backend 默认与 REDIS_URL 一致，支持 rediss 时自动补全 `ssl_cert_reqs`。
- **任务模块**：`include=["app.tasks"]`，主要任务在 `app/tasks/kb_tasks.py`：
  - `kb.add_files`：添加文件到知识库（切分 + 向量化）；
  - `kb.reindex_file`：单文件重新索引；
  - `kb.reindex_all`：整库重新索引。
- 任务内通过 `create_async_engine_and_session_for_celery` 获取 DB 会话，执行完后 `engine.dispose()`；结果与状态由前端轮询 `/api/v1/tasks/{task_id}` 获取。

## 六、API 与依赖

### 6.1 认证与依赖（app/api/deps.py）

- **认证**：`get_current_active_user`（在 auth 模块）从 Bearer Token 解析用户并校验是否激活。
- **限流依赖**：  
  - `require_upload_rate_limit`：上传接口；  
  - `require_chat_rate_limit`：对话接口；  
  - `require_search_rate_limit`：检索接口。  
  超出时返回 429，并带每日/QPS 上限提示。
- **工具方法**：`get_client_ip(request)`，支持 `X-Forwarded-For`。

### 6.2 认证 API（/api/v1/auth）

- 注册、登录（Form：username/password）、获取当前用户、修改密码等；登录成功更新 `User.last_login_at`，并签发 JWT。

### 6.3 问答 API（/api/v1/chat）

- **GET /settings/chat-attachment**：返回附件数量/大小/类型限制，供前端校验。
- **POST /completions**：同步问答，支持 `conversation_id`、`knowledge_base_id`、开关（RAG、MCP 工具、Skills 工具）、附件列表。
- **POST /attachments/upload**：上传即解析（图片走 LLM 解析/OCR，视频抽帧描述，其他走文本提取），结果写入 Redis，返回 `upload_id`；发消息时在 attachments 中传 `upload_id` 或 base64。
- **POST /completions/stream**：流式问答，请求体 JSON（可含 attachments、knowledge_base_ids 等），SSE 返回 token 与最终 done（含 conversation_id、confidence、sources、web_sources 等）。
- **GET/POST /conversations**：列表分页，带 Redis 缓存。
- **GET /conversations/{conv_id}`**：会话详情（含消息），带缓存。
- **GET /conversations/{conv_id}/messages**：仅消息列表。
- **DELETE /conversations/{conv_id}`**：删除会话并清理相关缓存。

### 6.4 检索 API（/api/v1/search）

- **POST /images**：以文搜图，可选单个/多个知识库，top_k 限制。
- **POST /unified**：多模态统一检索，传 `query` 或 `image_base64`，返回文档块与图片结果。
- **POST /by-image**：图搜图（body 中 base64）。
- **POST /by-image/upload**：图搜图（文件上传）。

### 6.5 知识库、文件、任务、仪表盘、计费、审计、MCP、Steward、Bash

- 知识库：CRUD、添加/移除文件、触发异步添加与重索引、检索（含以文搜图/图搜图/统一检索）由 `KnowledgeBaseService` 实现。
- 文件：上传（含安全校验、MinIO）、列表、删除，与知识库解耦；上传可走限流依赖。
- 任务：提交异步任务后返回 task_id，前端轮询 GET /tasks/{task_id} 获取状态与结果。
- 仪表盘：统计与用量快照（可走 Redis 缓存）。
- 计费：套餐、订阅、订单、发票等。
- 审计：按用户/时间/操作类型等查询 audit_logs。
- MCP：服务配置的增删改查，工具列表与执行在 Chat 流程中通过 `mcp_client_service` 动态拉取与调用。
- Steward / Computer Steward：浏览器自动化与桌面控制接口。
- Bash：命令审批（白名单、审批模式、过期时间）接口。

## 七、核心服务层

### 7.1 问答与 RAG（ChatService）

- **职责**：会话与消息持久化、RAG 检索、工具调用（MCP + Skills）、联网检索、对话历史（含可选总结）、流式/非流式回复。
- **RAG 流程**：  
  - 单知识库：`_rag_context`；多知识库：`_rag_context_kb_ids` 或 `_rag_context_all_kbs`。  
  - 多查询：原问 + 查询改写（`query_expand`）生成子问题。  
  - 向量检索（embedding + 向量库 search）+ 全文检索（关键词 LIKE + BM25 或关键词计数），按 RRF 融合。  
  - 可选 Rerank 重排，再按知识库配置做上下文窗口扩展（相邻块扩展）。  
  - 置信度低于阈值时在系统提示中说明，并可选返回检索上下文给前端展示。
- **工具阶段**：先组 MCP 与 Skills 的 OpenAI 格式工具列表，调用 `chat_completion_with_tools`；若有 tool_calls，则执行 MCP/Skills 并合并结果再继续生成。
- **联网**：`web_search_service.should_use_web_search` 启发式判断是否联网，`web_search` 使用 ddgs 多后端回退，结果格式化后注入上下文并写入 message 的 web_retrieved_context、web_sources。
- **附件**：用户消息支持多模态 content（文本 + 图片 URL/base64）；上传文件解析后以文本形式注入「用户上传的文件内容」；会话展示用 `attachments_meta`（含 dataUrl 等）存 LONGTEXT。
- **历史**：最近 N 条完整保留，更早的可选 LLM 总结后与最近 N 条一起拼成「对话历史」上下文；流式为降低首字延迟可跳过总结。

### 7.2 LLM 服务（llm_service）

- **接口**：OpenAI 兼容（AsyncOpenAI），base_url 与 api_key 来自配置，支持百炼等兼容端点。
- **能力**：  
  - `chat_completion`：单轮对话，可带系统提示与上下文（知识库+历史）；  
  - `chat_completion_stream`：流式逐 token；  
  - `chat_completion_with_tools`：支持 tool_calls，返回 content 与 tool_calls 列表；  
  - `query_expand`：查询改写/子问题生成（用于 RAG 多查询）。
- **系统时间**：`get_system_time_context()` 拼入 system content，便于模型感知当前时间。

### 7.3 嵌入与向量（embedding_service / vector_store）

- **Embedding**：使用阿里云百炼多模态 API（qwen3-vl-embedding），支持文本与图片同一向量空间；`get_embedding`、`get_embeddings`（批量，每批 20）、`get_embedding_for_image`。
- **向量库**：见 5.3；插入时使用 chunk 确定性 vector_id，便于按 chunk 列表删除与检索。

### 7.4 知识库服务（KnowledgeBaseService）

- **CRUD**：创建/更新/删除知识库，支持分块与检索相关配置。
- **文件与分块**：添加文件时从 MinIO 取回内容，按扩展名提取文本（PDF/Word/Excel/PPT/图片等）；扫描版 PDF 按页渲染后 OCR；可选敏感信息脱敏；按知识库或全局分块参数切分，生成 embedding 并写入向量库与 chunks 表；支持以文搜图、图搜图、多模态统一检索（向量 + 可选 BM25/RRF/rerank）。
- **删除知识库**：删除关联 chunk、向量库中对应向量、KnowledgeBaseFile 与知识库记录。
- **重新索引**：单文件或全库，先删旧 chunk 与向量，再重新切分与向量化。

### 7.5 Rerank 与 BM25

- **Rerank**：调用 DashScope text-rerank API，对 RAG 候选文档重排，返回带 relevance_score 的列表。
- **BM25**：`bm25_service` 对候选 (chunk, content) 做 BM25 打分，与向量检索结果经 RRF 融合；支持中文与英文 tokenization，简单停用词过滤。

### 7.6 联网检索（web_search_service）

- **触发**：`should_use_web_search(query, rag_has_content)` 启发式判断（时间词、技术词、短问句、RAG 无内容等）。
- **执行**：`ddgs` 多后端（duckduckgo/google/brave/bing）回退，结果归一化为 title/url/snippet，条数受 WEB_SEARCH_MAX_RESULTS 限制。
- **格式化**：`format_web_context` 供注入 LLM 上下文；原始结果写入 message 的 web_sources。

### 7.7 MCP 客户端（mcp_client_service）

- **职责**：解析 MCP 服务配置（含 mcpServers 解包、url/baseUrl、headers 中 `${VAR}` 环境变量替换），连接 stdio/SSE/Streamable HTTP 服务；列出工具、执行工具调用。
- **与 Chat 集成**：Chat 流程中从 DB 读取已启用 MCP 服务，通过 `gather_openai_tools_and_call_map` 得到 OpenAI 格式工具与调用映射，在 `_try_tool_phase` 中与 Skills 一起参与 tool_calls；另提供 `mcp_list_tools` 供用户问「有哪些 MCP 工具」时动态列出。

### 7.8 文件与安全

- **FileService**：上传校验（大小、类型、扩展名）、MD5、MinIO 存储、DB 记录；同 MD5 策略（use_existing/overwrite）；删除时清理 MinIO 与 chunks/向量。
- **file_security_service**：文件名长度与禁止扩展名、魔数校验（真实类型与扩展名一致）、可选病毒扫描（ClamAV socket）。
- **sensitive_mask_service**：入库与检索前对身份证、手机号等敏感信息打标或脱敏（可配置开关）。

### 7.9 审计、限流与缓存

- **audit_service**：关键操作（上传、删除知识库/文件、配置变更等）写入 audit_logs，带 user_id、action、resource_type/id、detail、ip、request_id。
- **rate_limit_service**：基于 Redis 的每日上传/每日对话计数、检索 QPS 计数；`get_usage_snapshot` 供仪表盘展示。
- **cache_service**：统一 Key 前缀与 TTL，封装 get/set/delete/delete_by_prefix；会话列表/详情、知识库列表/详情、仪表盘统计、智能问答上传临时数据等。

### 7.10 其他服务

- **ocr_service**：图片文本提取（可选 PaddleOCR 或 DashScope OCR 等）。
- **video_extract_service**：视频抽帧并用视觉模型描述，供智能问答附件与检索。
- **steward_tools / skill_loader**：Skills 工具注册与执行（如 skill_list、file_write 等），与 MCP 并列参与 Chat 的 tool_phase。
- **memory_service**：本地记忆（OpenClaw 风格），可选启用。
- **time_context**：系统时间上下文字符串，供 LLM 使用。
- **billing_service / user_service**：计费与用户信息维护。

## 八、安全与合规

- **认证**：JWT Bearer，密码 bcrypt 哈希；Token 过期时间可配置。
- **限流**：按用户的上传/对话/检索 QPS，超限 429；Redis 不可用时降级为放行。
- **文件安全**：类型与魔数校验、禁止可执行/脚本扩展名、可选 ClamAV；敏感信息脱敏可开关。
- **审计**：关键操作写 audit_logs，支持 request_id 链路追踪；可配置关闭。

## 九、健康检查与运维

- **GET /health**：返回 status（healthy/degraded）、dependencies 中 database/redis/vector/minio 各自的 ok 与 message。
- **日志**：Loguru，控制台 + 文件轮转（大小与保留天数、压缩），级别与路径来自配置。
- **请求 ID**：所有响应可带 X-Request-ID，便于与审计、日志关联。

## 十一、依赖与兼容性（requirements.txt 要点）

- **FastAPI/Starlette/Uvicorn**：与 Pydantic v2、sse-starlette 兼容；mcp>=1.15 要求 uvicorn>=0.31.1。
- **数据库**：SQLAlchemy 2.x、asyncpg、asyncmy、pymysql；Alembic 做迁移。
- **向量**：qdrant-client、pymilvus；pymilvus 依赖 pkg_resources，Python 3.12 需 setuptools<68；marshmallow 需 3.x（pymilvus 使用 __version_info__）。
- **AI**：openai 2.x、langchain/langchain-openai、sentence-transformers/transformers/torch（部分能力）；DashScope 通过兼容端点与 httpx 调用。
- **MCP**：mcp>=1.15、anyio 4.7、httpx-sse；Streamable HTTP 客户端用于阿里云/百炼等 MCP。
- **Windows**：main.py 与 run.py 中设置 `WindowsProactorEventLoopPolicy`，避免 Playwright 等 create_subprocess_exec 报错；marshmallow 与 pymilvus 的兼容性在 main 与 vector_store 中做了前置修复。

## 十二、小结

后端采用 **FastAPI + 异步 SQLAlchemy + 双向量库 + Redis + MinIO + Celery** 的清晰分层：API 层、服务层、数据与外部依赖（向量/存储/队列）。RAG 链路支持 **多查询、向量+全文、RRF、Rerank、BM25、上下文扩展**；问答支持 **多模态附件、MCP/Skills 工具、联网检索、会话历史与总结**。配置集中、可扩展（多知识库、多 MCP、多模型端点），安全与可观测性通过限流、审计、请求 ID 与健康检查覆盖。后续扩展可在此基础上增加新路由、新服务或新向量/存储后端，并沿用现有依赖注入与异步会话管理方式。
