# 资源模型与多租户缺口（改造 E-1）

> 对照现行 ORM 模型，说明「企业多租户」若上线还需补什么。

## 1. 现行模型（用户级隔离）

| 实体 | 表 / 关系 | 隔离方式 |
|------|------------|----------|
| 用户 | `users` | 认证主体 |
| 知识库 | `knowledge_bases.user_id` | **按用户**拥有 |
| 文件 | `files` → owner | 上传归属用户 |
| 分块 | `chunks.knowledge_base_id` | 随知识库 |
| 对话 | `conversations.user_id` | 随用户 |
| 评测集等 | `benchmark_datasets` 等 | 视实现，多为用户或全局 |

**结论**：当前是 **单租户产品形态**（每用户一套知识库），数据隔离依赖 `user_id` / `knowledge_base_id` 查询条件，**无独立「租户 Tenant」实体**。

## 2. 若演进为多租户（组织/团队）

| 能力 | 缺口 | 建议 |
|------|------|------|
| 组织维度 | 无 `tenants` / `organizations` 表 | 新增租户表，用户与租户多对多（`memberships`） |
| 知识库归属 | 仅 `user_id` | 增加 `tenant_id` 或 `org_id`，迁移策略：默认租户=个人 |
| 向量集合 | 多为全局 collection + metadata | 按租户拆分 collection 或强制 filter 字段 |
| 计费与配额 | `plans` / `usage` 偏用户级 | 按租户聚合用量（可选） |
